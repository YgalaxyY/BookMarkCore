## 1. Анализ ресурсов сайта и система фильтров

### Реализованная система фильтров
Разработана и внедрена новая система фильтрации (`filters.js`), которая работает на стороне клиента (Client-side) и предоставляет независимые инструменты фильтрации для каждой категории.

**Функциональные возможности:**
1.  **Автоматическое тегирование**: Скрипт анализирует заголовки, описания и контент карточек при загрузке страницы.
2.  **Секционные фильтры**: Вместо одного глобального фильтра, каждая категория (например, `#ai`, `#dev`) имеет собственную панель управления.
3.  **Умное отображение**: Выпадающие списки (Task Type, Complexity, Tech Stack) отображаются только если в данной категории есть соответствующие ресурсы.
4.  **Локальный поиск**: Поиск работает внутри конкретной секции, позволяя быстро находить инструменты по ключевым словам.
5.  **UI**: Панели фильтров интегрированы непосредственно в заголовки секций.

## 2. Анализ и оптимизация main компонента (`main.py`)

### Текущее состояние (Code Review)
Модуль `main.py` представляет собой монолитный скрипт, выполняющий функции Telegram-бота, веб-сервера (aiohttp) и менеджера контента.

**Проблемы:**
1.  **Архитектура**:
    *   Смешивание логики бота, бизнес-логики (анализ контента) и представления (генерация HTML).
    *   Отсутствие четкого разделения ответственности.
2.  **Производительность**:
    *   **Критично**: При каждом запросе `/ask` или добавлении ресурса бот скачивает полный файл `index.html` (5000+ строк) с GitHub, парсит его регулярными выражениями и (при записи) загружает обратно. Это создает огромную задержку и риск конфликтов версий.
    *   Синхронные вызовы в асинхронном коде (местами).
3.  **Надежность**:
    *   Парсинг HTML с помощью Regex (`re.search`, `re.sub`) крайне ненадежен. Любое изменение верстки сломает бота.
    *   Отсутствие базы данных. `index.html` используется как база данных, что является анти-паттерном для масштабируемых систем.
4.  **Безопасность**:
    *   Использование `ast.literal_eval` для парсинга JSON от нейросети безопаснее `eval`, но все же требует осторожности.
    *   Токены в `.env` (хорошо), но логика проверки админа простая.

### Рекомендации по оптимизации

#### 1. Архитектурный Рефакторинг
Разделить монолит на модули:
*   `bot.py`: Только обработка сообщений Telegram.
*   `core.py`: Бизнес-логика (анализ текста, работа с AI).
*   `storage.py`: Работа с данными (GitHub API, кеширование).
*   `utils.py`: Вспомогательные функции.

#### 2. Внедрение Базы Данных (или абстракции)
Вместо прямого редактирования HTML:
*   **Вариант А (Простой)**: Хранить данные в `data.json` на GitHub. Бот читает/пишет JSON. Сайт (`index.html`) при загрузке получает этот JSON и рендерит карточки (переход на чистый SPA).
*   **Вариант Б (Текущий, но оптимизированный)**: Кешировать `index.html` в памяти бота. Обновлять кеш только при успешном пуше или периодически.

#### 3. Оптимизация работы с GitHub
*   Не скачивать файл целиком каждый раз.
*   Использовать кеширование (TTL 5-10 минут) для функции поиска `/ask`.

#### 4. Улучшение Промптов
*   Структурировать промпты для AI, добавив четкие примеры JSON-ответа (One-shot learning), чтобы минимизировать ошибки парсинга.
*   Добавить поле `tags` в выходной JSON для лучшей фильтрации.
